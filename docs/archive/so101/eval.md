# VLA 模型评估 SOP（SO-101 + LeRobot）

## 1. 概述

本文档定义了基于 SO-101 机械臂、LeRobot 框架的 **微调后评估标准操作流程**。

评估分为两个阶段：

- **阶段一：训练指标预筛** — 通过 wandb 训练曲线判断训练是否正常收敛，筛选候选 checkpoint
- **阶段二：真机评估** — 在实际任务环境中部署候选 checkpoint，人工统计成功率

> **核心原则**：训练 loss 只能判断"训练没崩"，不能判断"模型好不好"。模型好不好，只有真机试了才知道。

---

## 2. 阶段一：训练指标预筛

### 2.1 训练收敛判断标准

通过 wandb 监控以下三个指标：

#### train/loss（Flow Matching Loss）

| 状态 | 判断标准 | 结论 |
|------|----------|------|
| ✅ 正常 | 从初始值快速下降，进入 0.05–0.4 范围内震荡的平台期 | 可以选 checkpoint 进入实机评估 |
| ⚠️ 需关注 | loss 平台期偏高（持续 > 0.5）但有缓慢下降趋势 | 可能需要更多训练步数或检查数据质量 |
| ❌ 异常 | loss 始终 > 1.0 不下降 | 数据格式/归一化/action space 配置有问题 |
| ❌ 异常 | loss 下降后突然爆炸（spike 到 10+） | 学习率过高，或数据中有异常样本 |
| ❌ 异常 | loss 降到 ≈ 0 | 严重过拟合，模型在死记硬背 |

> **注意**：π0/π0.5 的 flow matching loss 天然方差大，锯齿状波动是正常的，不必追求平滑曲线。80 个 episode 的小数据集波动会更明显。

#### train/grad_norm

| 训练模式 | 正常范围 | 异常信号 |
|----------|----------|----------|
| LoRA 微调 | 从 ~0 上升到 0.1–0.3 后稳定 | 持续上升不停，或突然 spike 到 10+ |
| 冻结 VLM，训练 Action Head | 初始较大（1.0–2.0），逐渐下降后震荡稳定 | 持续上升趋势（可能过拟合） |

#### train/lr

- 确认 lr schedule 形状正确（warmup → cosine decay）
- 确认 peak_lr 与配置一致

### 2.2 Checkpoint 选取策略

当 loss 进入平台期后（通常在 1k–2k 步之后）：

1. **每隔固定步数保存 checkpoint**（建议 500–1000 步）
2. **从平台期中选取 3–5 个 checkpoint** 作为候选
3. 选取原则：
   - 不要只选 loss 最低的（可能过拟合）
   - 不要只选最后一个（可能过拟合）
   - 在平台期的前段、中段、后段各选 1–2 个
   - 如果 grad_norm 后期有上升趋势，优先选上升之前的 checkpoint

### 2.3 预筛检查清单

在进入实机评估之前，确认以下事项：

- [ ] loss 已进入平台期，无持续上升趋势
- [ ] grad_norm 无爆炸或持续上升
- [ ] norm_stats 已正确计算且与训练数据匹配
- [ ] action space 维度与 SO-101 配置一致（6 DOF + 1 gripper = 7 维）
- [ ] prompt 文本与训练时使用的 prompt 一致
- [ ] 已选出 3–5 个候选 checkpoint

## 3. 阶段二：真机评估

### 3.1 评估环境准备

#### 3.1.1 物理环境标准化

为保证评估的可复现性，每次评估需固定以下条件：

- **桌面布局**：定义标准初始布局，拍照记录
- **光照条件**：保持一致的光照，避免逆光或强阴影
- **相机位置**：固定相机支架位置，不在评估过程中移动
- **物体集合**：使用固定的物体集合，记录每个物体名称和初始位置

#### 3.1.2 标准初始状态定义

```
【由 Claude Code 根据实际任务环境补充】
- 物体清单及摆放位置示意图
- 机械臂初始姿态（home position）
- 笔盒/容器的位置
```

### 3.2 评估任务定义

#### 任务示例：收纳桌面文具到笔盒

参考 π0.5 论文的评分 rubric 设计方式，将任务分解为可量化的子步骤：

| 评分项 | 描述 | 分值 |
|--------|------|------|
| 接近目标 | 机械臂末端移动到目标物体附近（距离 < 5cm） | +1 |
| 成功抓取 | 稳定抓取目标物体并抬起（离桌面 > 3cm） | +1 |
| 运输到位 | 将物体移动到笔盒上方（水平距离 < 5cm） | +1 |
| 成功放置 | 物体落入笔盒内部 | +1 |
| **单物体满分** | | **4 分** |

> 如果任务包含多个物体，则总分 = 4 × 物体数量。
> **任务进度 = 实际得分 / 总分 × 100%**

#### 通用评分原则（参考 π0.5）

- 每个子步骤独立打分，即使后续步骤失败，前面的得分保留
- 物体掉落到桌面以外的区域，该物体后续步骤不再得分
- 机械臂碰倒其他物体不扣分，但如果导致物体不可达则记录
- 超时（建议单物体 60 秒）后终止当前物体的尝试

### 3.3 评估执行流程

#### 3.3.1 单次评估 Episode 流程

```
1. 重置环境到标准初始状态（参照 3.1.2 的标准照片）
2. 确认相机画面正常、推理服务就绪
3. 发送任务 prompt（与训练时一致）
4. 启动策略推理，开始录制
5. 观察执行过程，按 rubric 实时打分
6. Episode 结束条件：
   a. 任务完成（所有物体收纳完毕）
   b. 超时（总时长超过设定上限）
   c. 明显失败且无法恢复（如机械臂卡住）
7. 记录得分、耗时、失败原因
8. 保存录制视频（用于复盘）
```

#### 3.3.2 评估批次设计

| 参数 | 建议值 | 说明 |
|------|--------|------|
| 每个 checkpoint 评估次数 | 10 次 | 参考 π0.5 论文每个任务 10 trials |
| 候选 checkpoint 数量 | 3–5 个 | 从训练平台期选取 |
| 评估间隔间环境重置 | 每次 | 确保初始状态一致 |
| 交叉评估 | 建议 | 不同 checkpoint 间穿插执行，控制环境变化 |

> **π0.5 论文做法**：在不同策略之间穿插执行评估（interleaving execution of policies），以控制环境随时间变化带来的影响。

### 3.4 评估记录模板

每次评估 episode 记录以下信息：

```markdown
## Episode Record

- **日期时间**：2026-03-27 14:30
- **Checkpoint**：step-3000
- **训练配置**：LoRA / freeze_vlm / ...
- **Episode 编号**：#3/10
- **任务 Prompt**："把桌上的文具收到笔盒里"
- **物体列表**：铅笔、橡皮、尺子
- **评分**：
  - 铅笔：接近(1) + 抓取(1) + 运输(1) + 放置(1) = 4/4
  - 橡皮：接近(1) + 抓取(1) + 运输(0) + 放置(0) = 2/4
  - 尺子：接近(0) + 抓取(0) + 运输(0) + 放置(0) = 0/4
- **总分**：6/12 = 50%
- **耗时**：87 秒
- **失败原因**：橡皮抓取后运输过程中掉落；尺子未尝试（超时）
- **视频文件**：eval_step3000_ep03.mp4
- **备注**：光照略暗于前两次
```

### 3.5 结果汇总与决策

#### 3.5.1 汇总指标

对每个 checkpoint 计算：

- **平均任务进度**（Average Task Progress）= 所有 episode 得分的均值
- **成功率**（Success Rate）= 完整完成任务的 episode 占比
- **标准差**（Std Dev）= 评估性能的稳定性

#### 3.5.2 Checkpoint 选择决策

```
如果 最佳 checkpoint 平均任务进度 > 70%:
    → 模型基本可用，可以进入优化迭代
如果 最佳 checkpoint 平均任务进度 在 30%–70%:
    → 模型有部分能力，需要分析失败模式并针对性改进
如果 所有 checkpoint 平均任务进度 < 30%:
    → 需要回到数据/训练阶段排查问题
```

#### 3.5.3 常见失败模式与对策

| 失败模式 | 可能原因 | 对策 |
|----------|----------|------|
| 机械臂完全不动或动作极小 | norm_stats 错误；action space 不匹配 | 检查 checkpoint 中的 norm_stats 和配置中的 action 维度 |
| 方向大致正确但精度差 | 数据量不足；数据质量差 | 增加高质量数据到 100–150 episode |
| 能抓但抓不稳 | gripper 控制数据不足 | 检查数据中 gripper action 的分布 |
| 偶尔成功但不稳定 | 正常现象，VLA 固有随机性 | 增加数据多样性 |
| 在训练场景能做，换物体/位置就失败 | 泛化不足 | 数据采集时增加物体/位置的多样性 |

## 4. 推理部署

真机评估依赖正确的推理部署。详细的推理部署流程见 [inference.md](./inference.md)。

评估阶段推理的关键要求：

- 推理延迟需满足实时控制需求（SO-101 控制频率参考 LeRobot 默认配置）
- 云端推理 + 笔记本控制电机的架构需确保网络延迟可控
- 推理服务需支持快速切换 checkpoint（用于对比评估）

```
【推理架构示意 — LeRobot gRPC 异步推理】

[GPU 服务器] ←gRPC→ [笔记本] ←USB→ [SO-101]
   Policy Server            Robot Client      电机控制
   π0.5 模型               控制循环
   推理服务                 观测采集+动作下发
```

详细推理部署流程见 [inference.md](./inference.md)。
